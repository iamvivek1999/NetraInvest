# ID Strategy — Hackathon-Safe Refinement
## MongoDB _id vs On-Chain campaignKey

> Final design decision before implementation begins.

---

## The Problem Being Solved

MongoDB `ObjectId` is a 12-byte value, represented as a 24-character hex string.  
Solidity `bytes32` is exactly 32 bytes.

Using ObjectId directly as bytes32 requires:
- Padding logic on the backend (12 bytes → 32 bytes)
- Trimming logic when reading back
- Risk of encoding mismatch between JavaScript and Solidity
- Debugging friction during a hackathon

**This is unnecessary complexity. Eliminate it with a clean two-ID design.**

---

## The Decision

### Use `bytes32` for the smart contract (not `string`)

**Comparison**:

| Criterion | `string campaignId` | `bytes32 campaignKey` |
|---|---|---|
| Gas cost as mapping key | Higher — dynamic type, more storage | Lower — fixed 32-byte slot |
| Solidity mapping syntax | `mapping(string => Campaign)` — works but requires `keccak256` internally | `mapping(bytes32 => Campaign)` — direct, efficient |
| ethers.js compatibility | Requires `ethers.utils.toUtf8Bytes()` encoding | Pass directly as hex string `0x...` |
| Risk of encoding mismatch | Higher — string length variations | Near zero — always 32 bytes |
| Human readability | Slightly better | Not needed (it's a key, not a label) |
| Hackathon safety | ⚠️ More friction | ✅ Clean and reliable |

**Verdict: `bytes32 campaignKey` wins on every dimension that matters for MVP.**

---

## The Two-ID Design

```
Every campaign has exactly two identifiers:

  _id          → MongoDB-generated ObjectId
                 Used for: all database operations, API routes, frontend URLs
                 Format: "664abc1234def567890abcde" (24-char hex string)
                 Scope: backend + frontend (API layer only)

  campaignKey  → backend-generated bytes32
                 Used for: all smart contract interactions
                 Format: 0x + 64 hex chars (32 bytes)
                 Scope: backend + smart contract + frontend (Web3 layer only)
```

---

## How campaignKey is Generated

**Generator**: Backend, at campaign creation time, before calling the contract.

**Method**: `ethers.utils.hexlify(ethers.utils.randomBytes(32))`

This produces a cryptographically random 32-byte value, e.g.:
```
0x7a3f9c2e1b4d8f0a6e5c3b2a9d7f4e1c8b5a2d9f6e3c0b7a4d1f8e5c2b9a6d3
```

**Why random bytes over a hash of ObjectId**:

| Option | Pros | Cons |
|---|---|---|
| Random bytes32 | No encoding logic. No dependency on ObjectId format. Instant. | Not deterministic (need to store it) |
| keccak256(ObjectId) | Deterministic — can re-derive if lost | Requires consistent encoding of ObjectId across JS and Solidity |
| ObjectId padded to bytes32 | Tied to DB key | Padding/trimming needed on every read/write |
| On-chain counter (uint256) | Auto-increments by contract | Backend must wait for tx receipt to get the ID before it can store it |

**Random bytes32 is the winner for hackathon**: zero encoding friction, generated in one line, stored once, used everywhere.

## Where Each ID Lives

```
MongoDB Campaign Document:
{
  _id:          ObjectId("664abc1234def567890abcde"),  ← MongoDB native
  campaignKey:  "0x7a3f9c2e1b4d8f0a...",              ← bytes32, stored as string
  title:        "AI Crop Monitor",
  ...
}
```

```
Smart Contract Storage:
{
  campaigns["0x7a3f9c2e1b4d8f0a..."] → CampaignData struct
}
```

**One-to-one relationship**: every MongoDB campaign document has exactly one `campaignKey`. They are stored together and linked for the lifetime of the campaign.

## How Each Layer Uses _id vs campaignKey

### MongoDB (Database Layer)
```
Uses: _id only

All queries use _id:
  Campaign.findById("664abc...")
  Campaign.findOne({ _id: req.params.id })

campaignKey stored as a plain string field.
Never used as a MongoDB query key.
```

### Backend (API + Contract Service Layer)
```
Uses: _id for DB queries, campaignKey for all contract calls

At campaign creation:
  1. Generate campaignKey = randomBytes32()
  2. Save campaign to MongoDB with both _id and campaignKey
  3. Call contract.createCampaign(campaignKey, ...)

At milestone release:
  1. Fetch campaign from MongoDB by _id → get campaignKey
  2. Call contract.releaseMilestone(campaignKey, milestoneIndex)

At investment verification:
  1. Read tx from chain (getTransaction)
  2. Verify tx.to === CONTRACT_ADDRESS
  3. Decode call data to confirm campaignKey matches

All API routes use _id in the URL:
  GET /api/campaigns/:id   → uses MongoDB _id
  PATCH /api/milestones/:id/approve  → uses MongoDB _id, then fetches campaignKey
```

### Frontend (UI + Web3 Layer)
```
Uses: _id for API calls, campaignKey for ethers.js contract calls

API calls use _id (from URL params or state):
  axios.get(`/api/campaigns/${campaignId}`)

Web3 calls use campaignKey (fetched from API response and stored in UI state):
  const campaign = await axios.get(`/api/campaigns/${id}`)
  const campaignKey = campaign.data.campaignKey

  await contract.invest(campaignKey, { value: amount })

Frontend never constructs or encodes campaignKey itself.
It receives it from the API and passes it through to ethers.js as-is.
```

### Smart Contract (Blockchain Layer)
```
Uses: bytes32 campaignKey only. Never knows about _id.

mapping(bytes32 => Campaign) public campaigns;
mapping(bytes32 => mapping(address => uint256)) public investments;

createCampaign(bytes32 campaignKey, ...)
invest(bytes32 campaignKey) payable
releaseMilestone(bytes32 campaignKey, uint256 milestoneIndex)
refund(bytes32 campaignKey)
```

## End-to-End Flow Walkthrough

```
STEP 1: Campaign Creation
─────────────────────────
Backend receives POST /api/campaigns from startup

  Backend:
    campaignKey = ethers.utils.hexlify(ethers.utils.randomBytes(32))
    
    Saves to MongoDB:
      { _id: <auto>, campaignKey: campaignKey, ... }
    
    Calls contract:
      contract.createCampaign(campaignKey, startupWallet, goal, pcts[])
    
    Returns to frontend:
      { _id: "664abc...", campaignKey: "0x7a3f9c...", ... }

STEP 2: Investor Browses Campaigns
────────────────────────────────────
Frontend: GET /api/campaigns
Backend returns array of campaigns, each with both _id and campaignKey

STEP 3: Investor Invests
─────────────────────────
Frontend has: campaignKey from the loaded campaign

  Web3 call:
    contract.invest(campaignKey, { value: ethers.utils.parseEther("10") })
  
  After tx.wait():
    POST /api/investments
    { campaignId: _id, campaignKey, txHash, amount, investorWallet }
  
  Backend:
    Verify txHash via getTransaction()
    Write investment to MongoDB (references campaign by _id)
    Update campaign.currentRaised

STEP 4: Milestone Release
──────────────────────────
Admin calls: POST /api/admin/milestones/:milestoneMongoId/approve

  Backend:
    Fetch milestone by _id → get milestoneIndex
    Fetch campaign by _id → get campaignKey
    
    Contract call:
      contract.releaseMilestone(campaignKey, milestoneIndex)
    
    After confirmation:
      Update milestone: { status: "released", releaseTxHash: txHash }
      Update campaign: { currentReleased += amount }
```

## Single Source of Truth Map

```
                     _id                    campaignKey
                  (ObjectId)                 (bytes32)
                      │                          │
                      │                          │
          ┌───────────▼──────────┐   ┌───────────▼──────────────┐
          │      MongoDB         │   │    Smart Contract         │
          │  Primary identifier  │   │  Primary identifier       │
          │  All DB queries      │   │  All contract calls       │
          │  All API URL params  │   │  Mapping key in Solidity  │
          └──────────────────────┘   └──────────────────────────┘
                      │                          │
                      └──────────┬───────────────┘
                                 │
                     ┌───────────▼──────────────┐
                     │   Campaign Document       │
                     │   (the bridge)            │
                     │                           │
                     │  _id:         "664abc..."  │
                     │  campaignKey: "0x7a3f..."  │
                     └───────────────────────────┘

Rule: Backend always starts from _id, fetches campaignKey when needed for contract.
Rule: Frontend always stores campaignKey in component state when on a campaign page.
Rule: Contract never knows about _id. MongoDB never keys by campaignKey.
```

## Summary

| Question | Answer |
|---|---|
| What type for on-chain key? | `bytes32` |
| How is campaignKey generated? | Backend: `ethers.utils.hexlify(ethers.utils.randomBytes(32))` |
| When is it generated? | At campaign creation, before contract call |
| Where is it stored? | MongoDB campaign document as a string field |
| Who generates it? | Backend only — frontend never constructs it |
| Who uses _id? | MongoDB queries, API route params, frontend navigation |
| Who uses campaignKey? | ethers.js contract calls (frontend), contract service (backend) |
| Is there encoding friction? | None — random bytes32 → pass through as hex string |
| Can this break during hackathon? | Extremely unlikely — no complex logic, just field passing |
